[FINDING 1 — CRITICAL — FIXED]
Title: OPM_REGISTRATION_KEY set + empty DB → unauthenticated mode (auth bypass)

File: src/open_project_manager_mcp/server.py — _check_auth()

Description:
  _check_auth enters unauthenticated mode when (a) OPM_TENANT_KEYS is not set
  and (b) the tenant_keys table is empty. It did NOT check whether
  OPM_REGISTRATION_KEY was set. An admin who enables self-service registration
  by setting OPM_REGISTRATION_KEY but does not pre-provision any OPM_TENANT_KEYS
  starts the server in an unauthenticated state — every request is allowed
  without a Bearer token until the first squad self-registers. This is the
  opposite of the admin's intent.

Impact:
  Full REST API access to all task data by any caller, for an indefinite window
  (the entire period before the first squad registers). On a LAN-exposed server
  this is a meaningful data disclosure and manipulation risk.

Fix applied:
  Changed the unauthenticated-mode gate from:
    if not tenant_keys:
        ...if not has_db_keys: return "system", None
  to:
    if not tenant_keys and not os.environ.get("OPM_REGISTRATION_KEY"):
        ...if not has_db_keys: return "system", None
  Unauthenticated mode now requires ALL three conditions:
    1. OPM_TENANT_KEYS not set
    2. OPM_REGISTRATION_KEY not set
    3. tenant_keys table is empty
  Presence of OPM_REGISTRATION_KEY signals auth intent; an empty DB means
  "no valid keys yet", not "open to everyone".

Regression test added:
  tests/test_registration.py::TestDbTokenAuth::test_auth_required_when_only_registration_key_set

[FINDING 2 — MEDIUM — FIXED]
Title: Rate limiter dict grows without bound (memory DoS)

File: src/open_project_manager_mcp/server.py — _check_rate_limit()

Description:
  _reg_attempts is a defaultdict(list) keyed by client IP. On each call, old
  timestamps for the requesting IP are pruned — but IPs that never make a second
  request retain their stale list entries forever. Under a flood of unique source
  IPs (each making exactly one request), the dict accumulates O(n) entries with
  no eviction. Each entry is small (~5 floats + overhead) but the count is
  unbounded.

Impact:
  Slow memory exhaustion over time on a long-running server exposed to varied
  source IPs (e.g., proxied behind a load balancer). Not a practical short-term
  risk for local-first deployments, but worth fixing for correctness.

Fix applied:
  At the start of _check_rate_limit, sweep the dict and delete any entry whose
  most-recent timestamp has already expired (i.e., the entire window for that IP
  has passed). Because registrations are infrequent, the O(n) sweep is
  negligible and keeps the dict size proportional to recently-active IPs only.

  Replaced the old append-then-prune pattern with a cleaner
  unexpired-list-first approach to avoid double-iteration.

[FINDING 3 — LOW — FIXED]
Title: DELETE /api/v1/register/{squad} — squad name not validated before DB query

File: src/open_project_manager_mcp/server.py — deregister_endpoint()

Description:
  The POST endpoint validates the squad name against _SQUAD_RE before touching
  the DB. The DELETE endpoint did not — it passed the raw path parameter
  directly to the SQL DELETE. While the query is parameterised (no SQL
  injection risk), an attacker with the registration key could send arbitrarily
  long or oddly-shaped strings as the squad segment. Starlette's {squad:str}
  path converter prevents path traversal (no slash), but defence-in-depth and
  API consistency both call for the same validation.

Impact:
  Low. Parameterised query prevents injection. No path traversal due to
  Starlette single-segment matching. Worst case: 404 response with unvalidated
  string echoed in the JSON error body ("Squad '...' not found."), which could
  cause garbled responses for non-UTF-8 or very long inputs.

Fix applied:
  Added _SQUAD_RE.match(squad) guard in deregister_endpoint, returning 400
  if the squad name is invalid, identical to the POST endpoint's behaviour.

[FINDING 4 — INFO — NO ACTION REQUIRED]
Title: OPM_REGISTRATION_KEY comparison timing safety

File: src/open_project_manager_mcp/server.py — register_endpoint(), deregister_endpoint()

Verification:
  register_endpoint: isinstance guard + hmac.compare_digest(provided_key, registration_key)
  deregister_endpoint: empty-string short-circuit (not reg_key_header) then
    hmac.compare_digest(reg_key_header, registration_key)
  Both paths are timing-safe for any non-empty submitted value. The empty-string
  short-circuit in deregister_endpoint is benign — an attacker trivially knows
  whether they sent the header or not; no oracle is created.
  Token not logged or included in error responses.  ✅ No issue.

Title: Rate limiter X-Forwarded-For spoofing

Verification:
  client_ip = request.client.host if request.client else "unknown"

[FINDING 6 — INFO — NO ACTION REQUIRED]
Title: tenant_keys plaintext storage

File: src/open_project_manager_mcp/server.py — _SCHEMA, INSERT INTO tenant_keys

Verification:
  Decisions.md §8 ("2026-04-01: Self-service token registration", item 7) explicitly
  documents the decision: store tokens plaintext, consistent with the existing
  local-first posture (env var keys, webhook secrets, --generate-token stdout all
  plaintext). Rationale: attacker with DB read access already has full task data;
  SHA-256 of a 32-byte random token adds no meaningful security; bcrypt explicitly
  rejected. Code matches documented decision.
  No token value appears in any log statement, error message, or activity_log row.
  ✅ Matches decisions.md.  No token leakage found.
