## Elliot: Telemetry + Push Notifications System

### Problem Statement

Andrew requested:
1. **Per-tenant telemetry** — Track activity/completions per squad to measure team throughput over time
2. **Push notifications to coordinators** — Real-time alerts when tasks are updated/completed

Andrew's specific question: *"when the MCP server is setup so that the connected clients are notifiable — is that the SSE? if so we need a way to keep track of everyone who needs notified."*

### Architecture Decision: 3-Tier Notification System

Designed hybrid model supporting three notification channels:

```
TIER 1: SSE Event Stream
  - Real-time delivery to connected clients
  - Zero latency, no polling
  - Requires active connection

TIER 2: Webhook Delivery
  - Guaranteed async delivery via HTTP callbacks
  - Requires coordinator to run HTTP server
  - Already implemented

TIER 3: Message Queue (Persistent/Pull-based)
  - Messages stored in DB with TTL
  - Coordinator polls via MCP tool
  - Works when offline
```

### Database Schema

**Telemetry Metrics Table:**
- `telemetry_metrics` — Hourly activity buckets per squad
  - Fields: squad, metric_type, project, count, period_start, period_end
  - Indexed: (squad, metric_type, project, period_start)
  - Metrics tracked: task.created, task.completed, task.updated, task.deleted

**SSE Client Registry:**
- `sse_clients` — Per-connection client metadata
  - Fields: id (UUID), squad, connected_at, last_seen, filter_event, filter_squad, user_agent
  - DB-backed (survives restart), queryable, supports distributed deployment
  - Replaces anonymous in-memory `_event_bus_clients` list

**Pending Notifications:**
- `pending_notifications` — Persistent message queue
  - Fields: id, recipient (squad), event_type, payload (JSON), created_at, expires_at (TTL=7d), read_at
  - Indexed: (recipient, read_at), (expires_at)

### New MCP Tools (v0.3.0)

| Tool | Purpose |
|------|---------|
| `get_my_telemetry(metric_type, project, hours, limit)` | Squad retrieves own telemetry |
| `get_squad_telemetry(squad, metric_type, hours, limit)` | Admin retrieves specific squad telemetry |
| `list_connected_clients(squad)` | Admin lists currently connected SSE clients |
| `list_pending_notifications(event_type, unread_only, limit)` | Squad retrieves pending notifications |
| `acknowledge_notifications(notification_ids)` | Squad marks notifications as read |

## Dom: Project-Level Data Privacy & Permissions System

### Problem Statement

Currently, any squad with a valid bearer token can read/write tasks in **any** project. No access control exists. High-severity security gap.

### Threat Model

| Threat | Severity |
|--------|----------|
| Cross-squad data access — fsociety can read/modify mrrobot's tasks | HIGH |
| Project squatting — Squad A creates project before Squad B can claim it | MEDIUM |
| Task injection — Malicious squad creates misleading tasks in another's project | HIGH |
| Reconnaissance — Any squad can enumerate all projects | MEDIUM |
| Notification leakage — Webhooks fire to unauthorized recipients | MEDIUM |
| Activity log exposure — get_activity_log reveals all projects' history | MEDIUM |

### Design Decision: Per-Project ACL with 3 Roles

```
owner       → full control (grant/revoke, delete project)
contributor → create/update/delete tasks
reader      → read-only access
```

**Key Principles:**
1. Projects must be explicitly created (no implicit creation via `create_task`)
2. Creator = initial owner
3. Default: private (accessible only to owner until grants made)
4. Multiple owners allowed (prevents orphan projects)
5. Token = identity (bearer token maps to squad, squad maps to ACL)

### Database Schema

**Projects Registry:**
- `projects` — Explicit project records
  - Fields: id, name, description, created_by, created_at, updated_at
  - Indexes: created_by

**Project Access Control:**
- `project_access` — ACL per project
  - Fields: project_id, squad, role (owner|contributor|reader), granted_by, granted_at
  - Primary key: (project_id, squad)
  - Indexes: project_id, squad

**Access Control Helper:**
- `_check_project_access(squad, project_id, required_role)` — Gating function
  - Returns (access_granted: bool, error: Optional[str])
  - Used in all project-scoped operations

### New MCP Tools (v0.3.0)

| Tool | Purpose | Role Required |
|------|---------|---------------|
| `create_project(id, name, description)` | Create project (caller becomes owner) | N/A |
| `get_project(project_id)` | Retrieve project metadata | reader+ |
| `list_my_projects()` | List projects caller has access to | N/A |
| `grant_project_access(project_id, squad, role)` | Grant/modify squad's access | owner |
| `revoke_project_access(project_id, squad)` | Revoke squad's access (not self) | owner |
| `delete_project(project_id)` | Delete project (cascade to tasks) | owner |
| `list_project_members(project_id)` | List all squad members + roles | reader+ |
| `get_project_access_level(project_id)` | Retrieve caller's access level | reader+ |

### Migration Strategy (v0.3.0)

1. **Deny-all default** — New code gated by `_check_project_access()`
2. **Existing tasks** — Assign to "default" project (created by system)
3. **Feature flag** — Enable project access checks gradually (per-endpoint)
4. **Backward compatibility** — Unprotected endpoints available during transition

### Build Order (BO-16 to BO-23)

1. **BO-16: Create projects table** — Schema, `create_project` tool
2. **BO-17: Create project_access table** — ACL schema, `_check_project_access()` helper
3. **BO-18: Project query tools** — `get_project`, `list_my_projects`, `list_project_members`
4. **BO-19: Project access management** — `grant_project_access`, `revoke_project_access` tools
5. **BO-20: Project deletion** — `delete_project` tool (cascade to tasks)
6. **BO-21: Gate create_task** — Require project + read access
7. **BO-22: Gate list_tasks** — Filter to accessible projects only
8. **BO-23: Migrate existing data** — Create "default" project, assign all existing tasks

**Dependencies:** BO-17→BO-16, BO-18→BO-17, BO-19→BO-17, BO-21→BO-19, BO-22→BO-19, BO-23→BO-22

**Estimated effort:** 3-4 days (Darlene implementation + Romero tests)
