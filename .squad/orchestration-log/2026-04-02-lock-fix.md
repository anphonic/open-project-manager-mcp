## Elliot: asyncio.Lock Starvation Root Cause Analysis

**Finding:** Problem is NOT SQLite write lock — it's **asyncio.Lock starvation**.

**Details:**
- Single asyncio.Lock guards ALL 23 write operations
- When session reaper calls transport.terminate(), if a task held _lock, Python does NOT auto-release it
- Lock stays acquired forever
- All subsequent write operations block indefinitely
- Reads work (don't use lock); writes hang (wait on lock)

---

## Darlene: 4-Change Implementation

### Change 1: WAL + busy_timeout Pragmas (server.py ~211)
Defense-in-depth for SQLite contention scenarios.

### Change 2: _locked_write() Helper on 23 Write Ops (server.py ~233+)
Timeout wrapper with asyncio.wait_for() (Python 3.10+ compatible).

### Change 3: Lock Reset in session_reaper (__main__.py ~249)
Reaper force-releases lock after terminating sessions.

### Change 4: Raised timeout_keep_alive (__main__.py ~532)
5s → 30s (session reaper handles orphans at app layer)
