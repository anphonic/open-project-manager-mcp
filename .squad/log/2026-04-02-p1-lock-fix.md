# Session Log: P1 asyncio.Lock Starvation Fix
**Date:** 2026-04-02T22-46-34Z

## Summary

P1 production bug diagnosed and fixed: asyncio.Lock starvation in OPM server blocking all write operations.

## Diagnosis

Root cause: NOT SQLite write lock — single asyncio.Lock guards 23 write operations. When session reaper terminates abandoned sessions, if a task held the lock, Python does not auto-release it. Lock remains acquired indefinitely, blocking all subsequent writes.

Evidence:
- Single _lock = asyncio.Lock() on line 231 in server.py
- All 23 write ops use async with _lock
- No WAL or busy_timeout pragmas configured
- Symptom pattern: reads work (don't use lock), writes hang (wait on lock)

## Implementation

Four changes deployed:

1. **WAL + busy_timeout** — Defense-in-depth SQLite resilience
2. **_locked_write() timeout wrapper** — 30s timeout on all 23 write ops
3. **Lock reset in session_reaper** — Force-release after session termination
4. **timeout_keep_alive 5s → 30s** — Reduced overhead; session reaper handles orphans
