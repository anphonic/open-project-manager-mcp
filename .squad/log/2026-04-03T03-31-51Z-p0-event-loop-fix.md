## Incident Context

SKS team reported OPM server becomes unresponsive under load. HTTP GET requests hang when MCP clients perform write operations. Root cause: all sqlite3 operations block the asyncio event loop.

---

## Root Cause Confirmed

**Problem:** All 100+ sqlite3 calls (both reads and writes) are synchronous, blocking the entire event loop.

**Evidence:**
- `get_task()` tool: `conn.execute(...).fetchone()` inside non-async function
- REST `/tasks` endpoint: `conn.execute(...).fetchall()` inside async function (still blocking!)
- `_verify_bearer()` auth: bearer token DB lookup blocks event loop

**Impact:** While any sqlite3 call runs, ALL other requests (HTTP, WebSocket, SSE) are stalled.

---

## Solution Implemented

**Approach:** Wrap ALL sqlite3 calls with `asyncio.to_thread()` to move blocking operations to thread pool.

### Changes Made

1. **Created async helpers:**
   ```python
   async def _db_execute(query, params):
       return await asyncio.to_thread(lambda: conn.execute(query, params).fetchall())
   
   async def _db_execute_one(query, params):
       return await asyncio.to_thread(lambda: conn.execute(query, params).fetchone())
   ```

2. **Updated `_locked_write()`:**
   ```python
   async def _locked_write(write_fn):
       await asyncio.wait_for(_lock.acquire(), timeout=30.0)
       try:
           return await asyncio.to_thread(write_fn)  # KEY: thread pool
       finally:
           _lock.release()
   ```

3. **Converted 28 MCP tools to async def**
4. **Updated 14 REST handlers to use async helpers**
5. **Made `_verify_bearer()` async for auth**
