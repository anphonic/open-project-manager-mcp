FINDINGS

ISSUE 1 — socket.getaddrinfo() blocking the asyncio event loop
Severity: Medium
Status: FIXED

Analysis:
  _check_ssrf() called socket.getaddrinfo() — a synchronous, blocking syscall
  — directly inside an async function (register_webhook). Any DNS lookup that
  takes more than a few milliseconds (slow resolver, non-existent domain,
  timeout) would freeze the entire asyncio event loop, preventing all other
  tasks from running for the duration of the lookup.

Fix applied:
  Made _check_ssrf() async. The DNS call is now dispatched to the default
  executor via:
    addrinfos = await loop.run_in_executor(None, socket.getaddrinfo, hostname, 443)
  Updated register_webhook to await _check_ssrf(url).

--------------------------------------------------------------------------------

ISSUE 2 — IPv4-mapped IPv6 addresses bypass RFC1918 SSRF blocklist
Severity: High
Status: FIXED

Analysis:
  Python's ipaddress module treats IPv4-mapped IPv6 addresses (e.g.
  ::ffff:10.0.0.1) as IPv6Address objects. Comparing an IPv6Address against an
  IPv4Network (like 10.0.0.0/8) returns False — different address families are
  never considered contained. A resolver that returns ::ffff:10.0.0.1 instead
  of 10.0.0.1 would pass the blocklist check and allow SSRF into RFC1918.

Fix applied (two parts):
  1. In the per-address loop, after building the ip object, the code now also
     extracts ip.ipv4_mapped for any IPv4-mapped IPv6 address and checks BOTH
     the IPv6 form and the underlying IPv4 form against all blocked networks:

       ips_to_check = [ip]
       if isinstance(ip, ipaddress.IPv6Address) and ip.ipv4_mapped:
           ips_to_check.append(ip.ipv4_mapped)
       for check_ip in ips_to_check:
           for net in _SSRF_BLOCKED_NETWORKS:
               if check_ip in net: ...

  2. Added fc00::/7 (IPv6 Unique Local Addresses — RFC 4193, the IPv6
     equivalent of RFC1918) to _SSRF_BLOCKED_NETWORKS.

ISSUE 3 — DNS rebinding: SSRF check is registration-time only
Severity: Medium (documented, no code change)
Status: Documented — needs Elliot's review for decision

Analysis:
  _check_ssrf() runs only at register_webhook() time. A crafted domain whose
  DNS TTL expires during the check (or which uses low-TTL DNS rebinding) can:
    1. Resolve to a public IP at registration time → passes check.
    2. Be updated to resolve to 127.0.0.1 or a cloud metadata endpoint.
    3. Trigger the webhook to fire — now hits the internal address.

  This is the classic DNS rebinding attack against SSRF blocklists.

Mitigations already in place (partial defense):
  - The webhook fires over HTTPS; TLS certificate validation would reject a
    self-signed cert on an internal service. Most internal endpoints (metadata
    services, local admin panels) don't have valid certs — httpx will raise an
    error and the request silently fails.
  - fire-and-forget delivery: the response is discarded, so even on a miss
    no data is exfiltrated back to the attacker.

Recommended fix (not applied — requires Elliot's call):
  Re-resolve and re-validate the hostname immediately before each webhook
  delivery in _fire_webhooks(). This adds latency per webhook fire but closes
  the rebinding window. Alternatively, require HTTPS + enforce certificate
  pinning or use a fixed outbound proxy.

ISSUE 4 — No request-body size limit on REST API endpoints
Severity: High
Status: FIXED

Analysis:
  POST /api/v1/tasks and PATCH /api/v1/tasks/{id} called await request.json()
  with no body-size cap. An attacker (or buggy client) could send a multi-GB
  request body; Starlette's request.body() would buffer the entire body in
  memory before parsing, causing OOM or extreme latency.

Fix applied:
  Added module-level constant _MAX_REST_BODY = 1_048_576 (1 MiB).
  Added _read_json_body(request) helper inside _build_rest_router():
    - Checks Content-Length header first; returns 413 immediately if > 1 MiB.
    - Reads the body; returns 413 if actual size > 1 MiB.
    - Parses JSON; returns 400 on parse error.
  Both POST tasks_endpoint and PATCH task_endpoint now call _read_json_body
  instead of await request.json().

ISSUE 5 — No request-body size limit in _FixArgumentsMiddleware
Severity: Medium
Status: FIXED

Analysis:
  _FixArgumentsMiddleware in __main__.py intercepts all HTTP POST requests and
  reads the full body into memory (b"".join(chunks)) with no size cap before
  forwarding to the underlying ASGI app. A 1 GB body would be fully buffered
  regardless of which endpoint it targets.

Fix applied:
  Added module-level constant _MAX_MCP_BODY = 6_000_000 (6 MB). The limit is
  set above 5 MB to accommodate import_tasks payloads (the largest legitimate
  MCP tool call body) while blocking unreasonable inputs.
  The chunk-accumulation loop now tracks total bytes:
    - If total_size > _MAX_MCP_BODY: sends a 413 JSON response directly and
      returns without forwarding to the app.

ISSUE 6 — FTS5 error message leaked search engine details
Severity: Low
Status: FIXED

Analysis:
  On invalid FTS5 query syntax, search_tasks returned:
    "Error: search failed: {exc}"
  where exc was the raw sqlite3.OperationalError string, e.g.:
    "Error: search failed: fts5: syntax error near '('"
  This confirms to a caller that FTS5 is in use and reveals its query syntax
  constraints — useful fingerprinting for a more targeted attack.

Fix applied:
  Changed to a generic message that gives no engine details:
    "Error: search failed — invalid query syntax"

--------------------------------------------------------------------------------

ISSUE 7 — Tags not individually length-capped in bulk operations or single ops
Severity: Medium
Status: FIXED

Analysis:
  create_task, update_task, create_tasks, and update_tasks all accepted
  unbounded tag arrays. A caller could pass 1,000 tags of 1 MB each, bypassing
  _MAX_DESCRIPTION and storing gigabytes across tags fields. The v0.1.0 audit
  noted this as a follow-up item; it remained unaddressed in v0.2.0.

Fix applied:
  Added module-level constants:
    _MAX_TAG_LENGTH = 100  (max characters per tag string)
    _MAX_TAGS_COUNT = 50   (max number of tags per task)

  Added _validate_tags() helper that checks both limits and also verifies each
  tag is a string. Called in:
    - create_task (MCP tool)
    - update_task (MCP tool)
    - create_tasks (bulk, per-item)
    - update_tasks (bulk, per-item)
    - REST POST /api/v1/tasks
    - REST PATCH /api/v1/tasks/{id}

--------------------------------------------------------------------------------

ISSUE 8 — complete_tasks: individual IDs not length-capped
Severity: Low
Status: FIXED

Analysis:
  complete_tasks (bulk) accepted up to 50 task IDs with no per-ID length check.
  While the 6 MB middleware cap now limits total payload, in stdio transport
  (no middleware) an attacker could send IDs of unbounded length and cause
  oversized SQL parameters.

Fix applied:
  Added a pre-loop validation pass in complete_tasks: if any task_id in ids
  exceeds _MAX_SHORT_FIELD (500 chars), the entire call returns an error with
  the index of the offending item.

--------------------------------------------------------------------------------

ISSUE 9 — REST POST /api/v1/tasks accepted empty id and title
Severity: Low
Status: FIXED

Analysis:
  The REST POST handler used body.get("id", "") and body.get("title", "").
  _validate_create_params only checks max length, not empty strings. A POST
  with no "id" field created a task with id="" in the database, and a POST
  with no "title" created a task with title="". This corrupts the data model
  and the empty-id task could interfere with lookups.

Fix applied:
  Added explicit empty-string checks before _validate_create_params:
    if not tid: return 400 "Error: 'id' is required"
    if not body.get("title", ""): return 400 "Error: 'title' is required"

================================================================================
ITEMS NOT FIXED (addressed by design or need Elliot's review)
================================================================================

1. DNS rebinding (ISSUE 3 above) — CLOSED 2026-04-01: Elliot decision: keep registration-time check only; httpx verify=True now explicit; HTTPS/TLS cert validation is the practical mitigation.

2. FTS5 query power-user behavior — FTS5 accepts boolean operators, column
   filters, and phrase queries through the MATCH ? parameterized binding.
   Users can craft complex queries that are CPU-intensive. This is intentional
   FTS5 behavior, not an injection vector. Acceptable risk for single-tenant
   use; a future rate-limit on search_tasks would harden this.

3. REST API auth bypass in no-auth mode — when OPM_TENANT_KEYS is unset,
   _check_auth() returns "system" for all requests. This is by design
   (single-tenant, localhost-only default). No change required.

4. Activity log injection — all _log() call sites use hardcoded action strings
   and parameterized SQL. action, field, old_value, new_value cannot be
   injected via user input. Confirmed clean.
