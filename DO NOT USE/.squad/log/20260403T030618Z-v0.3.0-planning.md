## Key Decisions Made

### 1. Telemetry: Hourly Buckets Over Event Logging

**Decision:** Use `telemetry_metrics` table with hourly buckets instead of per-event logging.

**Rationale:**
- Efficient querying: 1 row per (squad, metric, hour) vs 1 row per event
- Bounded growth: Max 24 rows/day per metric vs unbounded
- Privacy-preserving: Aggregated counts, not individual event details
- Flexible aggregation: Roll up to daily/weekly at query time

**Metrics tracked:** task.created, task.completed, task.updated, task.deleted

### 2. Notifications: 3-Tier Hybrid Model

**Decision:** Support SSE (instant), Webhooks (guaranteed), and Message Queue (persistent) as complementary tiers.

**Rationale:**
- SSE alone doesn't work offline or for reconnection scenarios
- Webhooks require coordinator to run HTTP server (not always feasible)
- Message queue with DB persistence handles offline access and retry patterns
- Each tier serves different coordinator deployment models

**Client registry:** DB-backed `sse_clients` table (not anonymous in-memory list) to:
- Enable targeted notifications to specific squads
- Survive server restarts
- Support future distributed deployment
- Provide admin visibility into connected clients

### 3. Permissions: Per-Project ACL Over Global RBAC

**Decision:** Implement per-project ACL with 3 roles (owner, contributor, reader).

**Rationale:**
- Simpler than RBAC for small number of squads (<20)
- Explicit project creation (no implicit squatting)
- Default-private model (secure by default)
- Prevents privilege escalation via role confusion or default-open migration
- Supports multiple owners per project (prevents orphans)

### 4. Project Access Gating Strategy

**Decision:** Implement deny-all gating with gradual rollout per endpoint.

**Rationale:**
- New code uses `_check_project_access()` helper
- Existing tasks assigned to "default" project during migration
- Feature flags enable per-endpoint transition
- Backward compatibility maintained during rollout
- Prevents catastrophic failures from incomplete migration

## Known Constraints & Decisions

1. **Project strings → Project IDs:** Migrate from `project="string"` to `project_id=FK` reference. All existing data assigned to "default" project.

2. **Multiple owners required:** At least 1 owner must always exist. Cannot revoke last owner's access. Prevents orphaned projects.

3. **No implicit project creation:** `create_task` rejects nonexistent project IDs. Projects must be created first via `create_project` tool.

4. **Coordinator identity:** In notification flow, assume "coordinator" maps to project owner or explicit configuration. Out of scope for v0.3.0; handled in v0.3.1.

5. **Telemetry retention:** Hourly buckets retained indefinitely in v0.3.0. Pruning/rollup to daily in v0.3.1.

6. **Feature flags:** Implement `--enable-project-acl` flag to toggle permission checks. Allows gradual rollout.
